## Story

**Como** aluno da Grande Final (que completou o **ato Blindar** — grants + a demo-dinheiro do bypass — e viu sua compra atravessar os 5 microsserviços no F6),
**eu quero** receber um **Diploma vivo** que renderiza a telemetria **real** da minha própria sessão (meus correlation-IDs, a região onde meu sistema rodou, a hora exata do meu deploy) — assinado *"Squad AIOX + você"* — e enxergar num diagrama a topologia de segurança completa que eu mesmo blindei (VNet, Private Endpoint, Managed Identity, Key Vault, `X-Gateway-Key`),
**para que** eu tenha uma prova **infalsificável e pessoal** ("a Copa que você construiu, agora te responde") de que construí um sistema de nível produção — algo que quero compartilhar, não só um certificado genérico de conclusão.

## `[AUTO-DECISION]` — a pendência de design do épico, resolvida aqui

### Onde o Diploma vivo é servido: rota no frontend do lab (ESCOLHIDA) vs. página estática no Template Repo

**Decisão:** rota nova no frontend React **já existente** (`Lovable/World Cup Tickets Hub/`, deployado pelo aluno desde as Oitavas/Quartas), **não** uma página estática dentro do Template Repository `fifa2026-final-lab`.

**Por quê:**
- **A fonte de telemetria (App Insights/FlowEvents) exige autenticação e infraestrutura de API já existentes** no frontend atual (`AuthProvider`, `apiV2.ts`, o roteamento via gateway) — replicar isso do zero numa página estática no Template Repo seria reinventar a roda (e provavelmente inventar uma nova superfície de autenticação, contrariando Art. IV).
- **O próprio épico já aponta a branch:** a tabela de Stories lista "`phase-08-hardening` (Diploma React)" — código React implica o frontend existente, que vive nesta branch/repositório, não no Template Repo (que é infraestrutura de entrega, não código de aplicação).
- **"Surface no Template Repo" é satisfeito por link, não por duplicação:** o workflow `lab-grande-final.yml` (Story 4.5) já imprime URLs como padrão estabelecido (AC-11 da 4.5) — adicionar a URL do Diploma ao checkpoint final é suficiente para "aparecer" na experiência de entrega, sem duplicar código React num repositório de infraestrutura.

### Fonte concreta de telemetria: App Insights via FlowEvents (ESCOLHIDA) — não uma nova query SQL direta

**Decisão:** reusa `AppInsightsFlowEventRepository`/`Azure.Monitor.Query` (o mecanismo que já alimenta o F6), não uma nova consulta direta a `purchases` via SQL.

**Por quê:**
- **Reúso > invenção (Art. IV/IDS):** o F6 (EPIC-003 3.4) já resolve exatamente o problema de "telemetria real e correlacionável por aluno" — construir uma segunda fonte (SQL direto) duplicaria lógica e criaria duas versões da verdade sobre a mesma jornada de compra.
- **Consistência narrativa:** "o Diploma renderiza o estado final; o F6 já mostrou a jornada ao vivo" (EPIC-004 §Dependências — "4.6 é o último: o Diploma renderiza o estado final (telemetria do F6/EPIC-003)") — usar a MESMA fonte reforça que o Diploma é a "foto" do que o F6 já provou em movimento.
- **Alternativa registrada (não escolhida):** uma query SQL direta em `purchases` seria mais simples de implementar, mas duplicaria a fonte de verdade e não reusaria a MI/padrão já validado pela Story 4.1/F6 — rejeitada por essa razão.
